# Ford Focus OBD Dashboard

Ford Focus OBD Dashboard — это локальная веб-панель для отображения телеметрии автомобиля через OBD-II адаптер. Проект ориентирован на Ford Focus 2 Restyling с механической коробкой передач, но архитектура разделена на независимые слои, поэтому её можно адаптировать под другие автомобили, наборы PID и правила диагностики.

Приложение запускает Flask-сервер, подключается к OBD-II адаптеру, периодически считывает параметры двигателя и автомобиля, обогащает «сырые» значения прикладной логикой и показывает их в браузере в виде приборной панели.

## Назначение проекта

Проект помогает:

- наблюдать основные параметры автомобиля в реальном времени;
- видеть статус подключения OBD-адаптера и автомобиля;
- оценивать передачу, состояние сцепления и стиль езды;
- получать простые предупреждения по температуре, оборотам, нагрузке и пропускам зажигания;
- сохранять сессии подключения и телеметрию в локальную SQLite-базу для последующего анализа;
- расширять обработку данных через отдельные процессоры без переписывания UI и слоя чтения OBD.

## Возможности

### Онлайн-панель

Главная страница `/` отображает:

- спидометр;
- тахометр;
- температуру охлаждающей жидкости;
- текущий VIN, если он доступен через OBD;
- индикатор статуса подключения;
- оценку текущей передачи;
- состояние сцепления;
- throttle/load/intake/voltage;
- стиль езды;
- предупреждения;
- состояние цилиндров по признакам пропусков зажигания;
- коды DTC, связанные с misfire-диагностикой.

Фронтенд опрашивает endpoint `/data` и плавно анимирует значения на панели.

### Чтение OBD-II

`dashboard/obd_reader.py` отвечает за подключение к адаптеру и сбор данных. Сейчас считываются:

- скорость;
- обороты двигателя;
- положение дросселя;
- нагрузка двигателя;
- давление во впуске;
- напряжение бортовой сети или адаптера;
- VIN;
- уровень топлива;
- коды неисправностей DTC;
- температура охлаждающей жидкости.

Чтение температуры выполняется реже, чем realtime-параметры, чтобы не перегружать адаптер.

### Обогащение телеметрии

Папка `dashboard/vehicle_processors/` содержит независимые обработчики:

- `gear.py` — эвристическая оценка передачи по соотношению RPM/скорость;
- `clutch.py` — определение вероятно выжатого/поджатого сцепления;
- `driving_style.py` — оценка стиля езды и нагрузки;
- `shift.py` — рекомендации shift-light;
- `misfire.py` — разбор DTC P0301–P0304 и пометка подозрительных цилиндров;
- `alerts.py` — человекочитаемые предупреждения;
- `snapshot.py` — сборка итогового enriched-state для API `/data`.

### Сохранение данных

Пакет `collector/` находится в корне проекта рядом с `dashboard/` и отвечает за фоновое сохранение телеметрии:

- `collector/service.py` запускает периодический сбор состояния и управляет сессиями;
- `collector/storage.py` хранит данные в SQLite;
- `collector/migrations/` содержит SQL-миграции схемы.

По умолчанию база создаётся в `data/telemetry.sqlite3`. Папка `data/` находится в корне проекта, а файлы SQLite игнорируются Git, чтобы runtime-данные не попадали в репозиторий.

## Структура проекта

```text
.
├── collector/                  # Фоновый сборщик и SQLite-хранилище
│   ├── migrations/             # SQL-миграции схемы БД
│   ├── service.py              # Управление сессиями и периодическая запись
│   └── storage.py              # SQLite persistence layer
├── dashboard/                  # Flask-приложение и UI
│   ├── app.py                  # Точка входа Flask
│   ├── obd_reader.py           # Подключение и чтение OBD-II
│   ├── static/                 # CSS/JS панели
│   ├── templates/              # HTML-шаблоны
│   └── vehicle_processors/     # Обработка и обогащение телеметрии
├── data/                       # Локальные runtime-данные, включая SQLite-БД
└── README.md
```

## Требования

- Python 3.10+;
- OBD-II адаптер, доступный как последовательное устройство;
- автомобиль/ЭБУ с поддержкой стандартных OBD-II PID;
- Python-пакеты:
  - `Flask`;
  - `obd` / `python-OBD`.

На Linux пользователь, запускающий приложение, должен иметь доступ к последовательному порту адаптера, например `/dev/ttyUSB0` или `/dev/rfcomm0`.

## Быстрый запуск

### 1. Подготовить виртуальное окружение

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip
python3 -m pip install Flask obd
```

Если пакет `obd` недоступен в вашей среде, установите пакет библиотеки python-OBD тем способом, который используется в вашем окружении.

### 2. Подключить OBD-II адаптер

Подключите адаптер к автомобилю и компьютеру. Проверьте путь устройства, например:

```bash
ls /dev/ttyUSB* /dev/ttyACM* 2>/dev/null
```

Для Bluetooth-адаптера путь может отличаться, например `/dev/rfcomm0`.

### 3. Настроить переменные окружения

Минимально обычно достаточно указать порт адаптера:

```bash
export OBD_PORT=/dev/ttyUSB0
```

Доступные переменные:

| Переменная | Значение по умолчанию | Назначение |
| --- | --- | --- |
| `OBD_PORT` | `/dev/ttyUSB0` | Последовательный порт OBD-II адаптера |
| `OBD_REALTIME_INTERVAL` | `0.25` | Интервал чтения быстрых параметров, сек |
| `OBD_TEMP_INTERVAL` | `5` | Интервал чтения температуры ОЖ, сек |
| `OBD_RECONNECT_INTERVAL` | `2` | Пауза перед повторным подключением, сек |
| `TELEMETRY_DB_PATH` | `data/telemetry.sqlite3` | Путь к SQLite-базе |
| `TELEMETRY_SAVE_INTERVAL` | `1` | Интервал записи телеметрии в БД, сек |

### 4. Запустить приложение

Из корня проекта выполните:

```bash
python3 dashboard/app.py
```

Затем откройте в браузере:

```text
http://127.0.0.1:5000/
```

API с текущим состоянием доступен по адресу:

```text
http://127.0.0.1:5000/data
```

## Работа без автомобиля

Сейчас проект рассчитан на реальный OBD-II адаптер. Для разработки без автомобиля можно расширить `ObdReader`, добавив mock/simulation provider, который будет возвращать словарь состояния в том же формате, что и `get_state()`.

Минимальный набор полей состояния:

```python
{
    "speed": 0.0,
    "rpm": 0.0,
    "temp": 0.0,
    "throttle": 0.0,
    "load": 0.0,
    "intake": 0.0,
    "voltage": 0.0,
    "vin": "",
    "fuel_level": 0.0,
    "dtc": None,
    "reverse": False,
    "connected": False,
    "connection_status": "disconnected",
}
```

## Хранилище данных

SQLite-схема создаётся автоматически при старте `TelemetryStorage`. Миграции лежат в `collector/migrations/` и применяются один раз, с фиксацией в таблице `schema_migrations`.

Основные сущности:

- `vehicles` — автомобили по VIN;
- `sessions` — сессии подключения к автомобилю;
- `metrics` — измерения телеметрии, привязанные к сессии.

База по умолчанию находится в:

```text
data/telemetry.sqlite3
```

Если нужно перенести БД, задайте:

```bash
export TELEMETRY_DB_PATH=/path/to/telemetry.sqlite3
```

## Расширение проекта

### Добавление нового OBD-параметра

1. Добавьте чтение PID в `dashboard/obd_reader.py`.
2. Сохраните значение в `self.state`.
3. Если значение нужно отдавать на UI, оно автоматически попадёт в `/data` через `get_state()` и `enrich_state()`.
4. Добавьте отображение в `dashboard/templates/index.html` и `dashboard/static/js/dashboard.js`.
5. Если параметр должен сохраняться в БД, добавьте миграцию в `collector/migrations/` и обновите `collector/storage.py`.

### Добавление новой диагностической логики

1. Создайте отдельный модуль в `dashboard/vehicle_processors/`.
2. Реализуйте чистую функцию, принимающую нужные значения телеметрии.
3. Подключите её в `dashboard/vehicle_processors/snapshot.py`.
4. При необходимости экспортируйте функцию в `dashboard/vehicle_processors/__init__.py`.
5. Отобразите результат на UI или сохраните в БД.

Такой подход позволяет тестировать диагностические правила отдельно от Flask, OBD-адаптера и браузера.

### Адаптация под другой автомобиль

Для другого автомобиля обычно потребуется:

- изменить диапазоны в `gear.py` под передаточные числа и размер колёс;
- добавить manufacturer-specific PID для более точной диагностики;
- настроить набор предупреждений в `alerts.py`;
- расширить UI под дополнительные датчики;
- добавить миграции для новых сохраняемых метрик.

### Изменение UI

UI состоит из HTML-шаблона, CSS и JavaScript:

- `dashboard/templates/index.html` — структура панели;
- `dashboard/static/css/style.css` — внешний вид;
- `dashboard/static/js/dashboard.js` — опрос API, анимация и отрисовка приборов.

Endpoint `/data` возвращает JSON, поэтому можно создать альтернативный фронтенд, мобильный клиент или интеграцию с внешними системами без изменения OBD-слоя.

## Проверка проекта

Быстрая проверка синтаксиса:

```bash
python3 -m compileall dashboard collector
```

Smoke test хранилища можно выполнить отдельным Python-скриптом, создав временную БД и вызвав `start_session`, `save_metrics`, `end_session`.

## Ограничения

- Оценка передачи является эвристической и зависит от конкретной коробки передач, размера колёс и состояния сцепления.
- Generic OBD-II не предоставляет универсальные счётчики пропусков зажигания по цилиндрам, поэтому текущая логика опирается на DTC P0301–P0304 и простые признаки rough running.
- Не все автомобили и адаптеры поддерживают VIN, уровень топлива, напряжение и часть других PID.
- Приложение предназначено для локального запуска и не содержит production-настроек безопасности для публикации в интернет.

## Рекомендации по эксплуатации

- Не используйте панель так, чтобы она отвлекала водителя во время движения.
- Перед записью больших объёмов данных проверьте свободное место в `data/`.
- Для стабильной работы используйте качественный OBD-II адаптер и корректный serial-port.
- При настройке новых PID проверяйте, что адаптер и автомобиль действительно поддерживают эти команды.
